# OpenClaw provider capability surface audit (2026-04-14)

This report maps the provider-backed capability surface exposed by the local `openclaw@2026.4.8` package in `node_modules/openclaw`.

The goal is to answer a narrower question than “what can a plugin do?”:

- What **provider/media/search/embedding** surfaces can OpenClaw talk to today?
- Which of those are first-class user-facing surfaces versus plugin/runtime-only seams?
- What is the exact OpenClaw contract our NanoGPT plugin could target without changing OpenClaw core?

## Scope notes

- Included: provider-backed inference/media/search/embedding surfaces and the provider runtime hooks that shape them.
- Included: `openclaw infer ...` because it is the clearest headless user-facing surface for provider capabilities.
- Excluded: generic plugin APIs such as channels, hooks, tools, gateway methods, and services unless they directly affect provider capability ownership.

## High-level summary

- OpenClaw has first-class headless inference surfaces for `model`, `image`, `audio`, `tts`, `video`, `web`, and `embedding`. `node_modules/openclaw/docs/cli/infer.md:59-98`; `node_modules/openclaw/docs/cli/infer.md:109-118`
- OpenClaw's provider registration API is broader than the current infer command tree: it also supports `realtime transcription`, `realtime voice`, `music generation`, `web fetch`, and `memory embedding adapters`. `node_modules/openclaw/docs/plugins/sdk-overview.md:304-315`; `node_modules/openclaw/docs/plugins/sdk-overview.md:402-411`
- Text providers are more than “just call a model”: OpenClaw's `ProviderPlugin` contract includes catalog/discovery, dynamic model resolution, transport normalization, runtime auth exchange, usage reporting, catalog augmentation, and provider-owned embeddings. `node_modules/openclaw/dist/plugin-sdk/src/plugins/types.d.ts:966-1255`
- Important nuance: OpenClaw's docs label `registerSpeechProvider(...)` as “Text-to-speech / STT synthesis”, but the actual `SpeechProviderPlugin` type is synthesize-centric. File transcription lives under `MediaUnderstandingProvider.transcribeAudio`, and live STT lives under `RealtimeTranscriptionProvider`. `node_modules/openclaw/docs/plugins/sdk-overview.md:307-310`; `node_modules/openclaw/dist/plugin-sdk/src/plugins/types.d.ts:1510-1543`; `node_modules/openclaw/dist/plugin-sdk/src/media-understanding/types.d.ts:111-120`
- Embeddings are also slightly special: there is no top-level `registerEmbeddingProvider(...)`. OpenClaw exposes embeddings through `openclaw infer embedding create`, provider-owned `createEmbeddingProvider`, and memory-plugin-owned `registerMemoryEmbeddingProvider(adapter)`. `node_modules/openclaw/docs/cli/infer.md:118`; `node_modules/openclaw/docs/cli/infer.md:223-229`; `node_modules/openclaw/docs/plugins/sdk-provider-plugins.md:510`; `node_modules/openclaw/docs/plugins/sdk-overview.md:402-411`; `node_modules/openclaw/dist/plugin-sdk/src/plugins/types.d.ts:716`; `node_modules/openclaw/dist/plugin-sdk/src/plugins/types.d.ts:1177`

## Practical interpretation for this NanoGPT plugin

If the question is “what can OpenClaw talk to that a NanoGPT plugin could reasonably implement?”, the answer is:

- **Definitely first-class and user-facing today:** text/chat, image generation/edit, audio transcription, TTS, video generation, web search, web fetch, embeddings.
- **Supported by the plugin API even if not front-and-center in `infer`:** realtime transcription, realtime voice, music generation, memory embedding adapters, usage-reporting hooks.
- **Most important contracts for NanoGPT specifically:** text provider runtime hooks, web search provider, image generation provider, media understanding for STT, speech provider for TTS, and provider-owned embeddings.

That makes OpenClaw's current provider surface wider than this NanoGPT plugin's current implementation. The companion comparison report in `docs/openclaw-vs-nanogpt-provider-capability-coverage-2026-04-14.md` maps that gap directly.
